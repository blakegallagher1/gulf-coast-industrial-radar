## Failure modes we should handle in the adapter

1. **CAPTCHA verification failure:** Human checkbox requirement will block automated requests. Adapter must either:
   - Implement browser automation (Selenium/Playwright) to click checkbox
   - Use Commercial API subscription instead of web scraping
   - Handle 403/verification rejection gracefully

2. **Database update lag:** New filings may not appear for up to 24 hours. Adapter should implement retry logic with exponential backoff for recently filed entities.

3. **Similar name collisions:** "Louisiana businesses often use similar names with slight variations in spelling, punctuation, or entity designators." Results may return hundreds of matches. Adapter must implement fuzzy matching or require exact charter number for disambiguation.

4. **Merged/converted entities:** "Businesses that have merged with other companies or converted to different entity types may show complex filing histories." Adapter should parse "Mergers" and "Amendments On File" fields to reconstruct entity lineage.

5. **Incomplete search terms:** "Using too few keywords may return hundreds of results, while overly specific terms might miss relevant matches." Adapter should implement fallback search strategies (abbreviations, partial names, alternative spellings).

6. **Status-dependent access:** Dissolved/revoked entities may have limited or archived record availability. Adapter should handle gracefully.

7. **Session timeout:** No explicit session documentation. Assume standard HTTP session behavior; implement connection pooling and retry logic.
