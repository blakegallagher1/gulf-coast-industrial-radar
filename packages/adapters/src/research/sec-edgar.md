# Source: SEC EDGAR Submissions API + Filings

## Current production URLs

| Endpoint | URL | Purpose |
|----------|-----|---------|
| Submissions by CIK | `https://data.sec.gov/submissions/CIK{cik_padded}.json` | Filing history and metadata for a company |
| Company Facts (XBRL) | `https://data.sec.gov/api/xbrl/companyfacts/CIK{cik_padded}.json` | All XBRL financial facts for a company |
| Company Concept | `https://data.sec.gov/api/xbrl/companyconcept/CIK{cik_padded}/{taxonomy}/{tag}.json` | Specific XBRL data point across periods |
| XBRL Frames | `https://data.sec.gov/api/xbrl/frames/{taxonomy}/{tag}/{unit}/{period}.json` | Aggregated facts by calendrical period |
| Full-Text Search | `https://efts.sec.gov/LATEST/search-index?q={query}` | Filing search results |
| RSS Feed | `https://www.sec.gov/cgi-bin/browse-edgar?action=getcompany&CIK={cik}&type={form_type}&dateb=&owner=exclude&count=100&search_text=` | Real-time updates on newly submitted documents |

CIK values must be **zero-padded to 10 digits** in URL construction (e.g., Apple's CIK 320193 becomes `CIK0000320193`).

## Authentication & session model

**No authentication required.** All endpoints are public and do not require API keys.

**User-Agent header is mandatory.** The SEC requires a descriptive User-Agent string identifying the requester for rate-limiting and tracking purposes. Format: `<Company Name> <Contact Email>` or `<Name> [email protected]`.

Example:
```
headers = {'User-Agent': 'MyCompany [email protected]'}
```

## Endpoints / queries we use

### Submissions endpoint (primary for filing metadata)

**URL:** `https://data.sec.gov/submissions/CIK{cik_padded}.json`

**Parameters:** None (CIK embedded in URL path)

**Returns:** Complete filing history for a company, including:
- CIK, company name, entity type
- Array of filings with accession number, form type, filing date, report date, acceptance datetime
- Links to primary document and filing index

**Example request:**
```python
import requests

cik = '320193'  # Apple
cik_padded = str(cik).zfill(10)
url = f'https://data.sec.gov/submissions/CIK{cik_padded}.json'
headers = {'User-Agent': 'MyCompany [email protected]'}
response = requests.get(url, headers=headers)
filings = response.json()
```

### Company Facts endpoint (XBRL financial data)

**URL:** `https://data.sec.gov/api/xbrl/companyfacts/CIK{cik_padded}.json`

**Parameters:** None (CIK embedded in URL path)

**Returns:** Structured XBRL facts organized by taxonomy (us-gaap, ifrs-full, dei, etc.) and tag name, with units and historical values.

**Example request:**
```python
url = f'https://data.sec.gov/api/xbrl/companyfacts/CIK{cik_padded}.json'
response = requests.get(url, headers=headers)
facts = response.json()
gaap = facts['facts'].get('us-gaap', {})
revenues = gaap.get('Revenues', {}).get('units', {}).get('USD', [])
```

### Company Concept endpoint (single XBRL tag across periods)

**URL:** `https://data.sec.gov/api/xbrl/companyconcept/CIK{cik_padded}/{taxonomy}/{tag}.json`

**Parameters:**
- `taxonomy`: e.g., `us-gaap`, `ifrs-full`, `dei`
- `tag`: e.g., `Revenues`, `NetIncomeLoss`, `Assets`

**Returns:** Time series of a single fact across all filings for that company.

### XBRL Frames endpoint (cross-company aggregates)

**URL:** `https://data.sec.gov/api/xbrl/frames/{taxonomy}/{tag}/{unit}/{period}.json`

**Parameters:**
- `taxonomy`: e.g., `us-gaap`
- `tag`: e.g., `AccountsPayableCurrent`
- `unit`: e.g., `USD`
- `period`: e.g., `CY2019Q1I` (calendar year 2019 Q1 instantaneous), `CY2019` (annual)

**Returns:** Aggregated fact for each reporting entity that last filed closest to the requested period.

## Field map → target predicates

| Source Field | Target Predicate | Extraction Path |
|--------------|------------------|-----------------|
| CIK | `sec.cik` | `submissions.cik_str` or `companyfacts.cik_str` |
| Form Type | `sec.filing.{form}` | `submissions.filings.recent[].form` → emit `sec.filing.10-K`, `sec.filing.10-Q`, `sec.filing.8-K`, `sec.filing.20-F` as applicable |
| Accession Number | `sec.accession` | `submissions.filings.recent[].accessionNumber` |
| Filing Date | `sec.filing_date` | `submissions.filings.recent[].filingDate` (ISO 8601) |
| Report Date | `sec.report_date` | `submissions.filings.recent[].reportDate` (ISO 8601) |
| Primary Document | `sec.primary_document` | `submissions.filings.recent[].primaryDocument` |
| XBRL Facts | `sec.xbrl.{taxonomy}.{tag}` | `companyfacts.facts.{taxonomy}.{tag}.units.{unit}[].value` |

**Signal emission logic:**
- Iterate `submissions.filings.recent[]`
- For each filing where `form` is in {`10-K`, `10-Q`, `8-K`, `20-F`}, emit `sec.filing.{form}` with metadata (CIK, accession, filing_date)
- For each XBRL fact in `companyfacts.facts.us-gaap`, emit `sec.xbrl.us-gaap.{tag}` with value and period

## Pagination & rate limits

**Submissions endpoint pagination:**
- Response includes `filings.recent[]` (most recent filings, typically ~100 entries)
- Older filings available in `filings.files[]` array with separate index files
- To retrieve complete history, parse `filings.files[]` and fetch each historical index file separately

**Rate limiting:**
- **10 requests per second per IP address** (SEC standard)
- Implement exponential backoff on 429 (Too Many Requests) responses
- Respect `Retry-After` header if present

**Recommended adapter behavior:**
- Cache submissions responses for 24 hours
- Stagger requests across multiple IPs if bulk-scraping many CIKs
- Use conditional requests (If-Modified-Since) where supported

## Failure modes we should handle in the adapter

1. **Invalid CIK format:** CIK not zero-padded or non-numeric → returns 404. Validate and pad to 10 digits before request.

2. **Missing User-Agent header:** SEC may reject requests without User-Agent or return 403. Always include descriptive User-Agent.

3. **Rate limiting (429):** Implement exponential backoff (start 1s, max 60s) and respect `Retry-After` header.

4. **Incomplete XBRL data:** Some older filings lack XBRL facts. Check `isXBRL` flag in submissions response; skip Company Facts endpoint if 0.

5. **Accession number format inconsistency:** Accession numbers may appear with or without hyphens (e.g., `0000320193-24-000077` vs `0000320193240000077`). Normalize on ingestion.

6. **Primary document missing:** `primaryDocument` field may be null for certain filing types. Fall back to parsing `filings.files[]` index.

7. **Timeout on large responses:** Company Facts endpoint for mega-cap companies can return 50+ MB JSON. Implement streaming JSON parser and request timeout (recommend 30s).

8. **Stale data in frames endpoint:** XBRL Frames aggregates are updated on a delay. Do not assume real-time accuracy for cross-company comparisons.

9. **Form type variations:** Form types include amendments (e.g., `10-K/A`, `10-Q/A`). Decide whether to filter or tag separately in signal emission.

10. **IP blocking:** Aggressive scraping may trigger temporary IP blocks. Implement circuit breaker pattern and alert on repeated 403 responses.

## Open questions for the engineer

2. **XBRL taxonomy scope:** Should we ingest all taxonomies (us-gaap, ifrs-full, dei, srt) or focus on us-gaap for US-domiciled companies?

3. **Historical backfill strategy:** Submissions endpoint returns ~100 recent filings. For complete history, must we parse `filings.files[]` indices. What is the acceptable latency for historical data availability?

4. **Accession number deduplication:** Can the same accession number appear in multiple CIK records (e.g., co-filers)? Should we deduplicate on accession + CIK composite key?

5. **Primary document retrieval:** Should the adapter fetch and parse the actual HTML/XML primary document, or only extract metadata from the submissions API?

6. **XBRL unit normalization:** Facts may be reported in different units (USD, shares, etc.). Should we normalize to a canonical unit or preserve as-is?
